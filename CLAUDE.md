# Dragline — каталог б/у спецтехники из Кореи

## Что это
Сайт-каталог б/у экскаваторов, запчастей и спецтехники из Южной Кореи.
Покупатели — Узбекистан/Кыргызстан/СНГ + English-аудитория.
Сделка: я выкупаю технику у корейца через своё ИП/법인 и отправляю покупателю.
Сайт = витрина, НЕ маркетплейс с оплатой.

## Стек (фиксированный — не менять без явного запроса)
- HTML + CSS + Vanilla JS. БЕЗ React/Vue/фреймворков.
- Хостинг: GitHub Pages
- Данные: Google Sheets через SheetDB API (read-only)
- Фото: Cloudinary (URL в таблице)
- Курс валют: бесплатный API (KRW→USD), кэш в localStorage на 1 час,
  fallback — захардкоженная константа

## Языки (4)
ko / uz / ru / en — JSON-файлы в /locales, переключатель в шапке,
выбор хранится в localStorage, дефолт по браузеру, fallback en.

## ЗАПРЕЩЕНО на текущем этапе
- Регистрация/авторизация (продавцов и покупателей)
- Форма добавления объявлений на сайте
- Корзина, оплата, чекаут
- Расчёт доставки/растаможки в коде — только текст-заглушка +
  ссылки на Telegram/WhatsApp/Instagram

Все объявления добавляю я вручную через Google Sheets.

## Структура данных (лист "listings" в Google Sheets)
id, category (excavator/parts/other_machinery), brand, model, year, hours,
condition, compatible_models, price_krw, location_kr, photos (URL через
запятую), status (available/sold/reserved),
name_ko/name_uz/name_ru/name_en, desc_ko/desc_uz/desc_ru/desc_en

## Структура файлов
```
index.html      — каталог с фильтрами (категория/марка/цена)
listing.html     — страница лота (галерея, цена KRW+USD, контакты)
about.html       — "как это работает"
css/style.css
js/main.js        — загрузка SheetDB, рендер карточек
js/i18n.js        — переключение языков
js/currency.js    — конвертация валют
js/listing.js     — логика страницы лота
locales/{ko,uz,ru,en}.json
```

## Дизайн
Минимализм, индустриальная тема: тёмно-синий + оранжевый/жёлтый акцент.
Шрифт с поддержкой кириллицы+латиницы (Inter/Roboto), Noto Sans KR для
хангыля. Полностью responsive, мобильный приоритет.

## Правила работы (важно!)
1. Перед написанием кода — сначала план (какие файлы, что меняется),
   жду одобрения.
2. Точечные правки в одобренных файлах, не переписывать целиком без запроса.
3. Маленькие шаги: один шаг = одна проверяемая фича.
4. После каждого шага — что сделано и как проверить в браузере.
5. Если предлагаешь архитектуру отличную от описанной — кратко объясни
   почему и жди одобрения.

## Текущий этап
MVP с placeholder-данными (тестовые лоты прямо в JS-массиве) для проверки
дизайна/языков/фильтров. SheetDB подключаем после одобрения визуала.
